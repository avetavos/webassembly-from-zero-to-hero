WebAssembly ถูกออกแบบมาสำหรับเบราว์เซอร์ แต่คุณสมบัติของมัน (sandbox ที่ปลอดภัย, ความเร็วใกล้เคียง native, และความพกพาข้ามภาษา) ทำให้มีคุณค่าเท่ากันบนเซิร์ฟเวอร์, edge node, อุปกรณ์ embedded และระบบ plugin กุญแจที่ปลดล็อกทุกสภาพแวดล้อมเหล่านี้คือ **WASI** — WebAssembly System Interface

## ทำไมต้องรัน Wasm นอกเบราว์เซอร์?

ในเบราว์เซอร์ JavaScript engine คือ host ที่ให้สภาพแวดล้อมรันไทม์, จัดการหน่วยความจำ และเปิดเผย browser API นอกเบราว์เซอร์ไม่มี JavaScript engine, DOM หรือ `fetch` โมดูล Wasm ที่ต้องการอ่านไฟล์, เขียน stdout หรือเปิด socket ต้องการ host อื่นมาจัดหาความสามารถเหล่านั้น

WASI คือคำตอบที่เป็นมาตรฐาน: ชุดนิยาม system-call คล้าย POSIX ที่แสดงออกมาเป็น Wasm import รันไทม์ใดก็ตามที่ implement WASI สามารถรัน binary `.wasm` เดียวกัน — บน Linux, macOS, Windows หรือไมโครคอนโทรลเลอร์ขนาดเล็ก

## ความสัมพันธ์ระหว่าง host, runtime และ module

ไดอะแกรมแสดงโมเดลสามชั้น:

1. **Host** — ระบบปฏิบัติการหรือ cloud platform ที่ตัดสินใจว่าทรัพยากร (directories, sockets, environment variable) ใดที่รันไทม์จะส่งผ่านได้
2. **WASI Runtime** — executable อิสระเช่น `wasmtime` หรือ `wasmer` ที่ implement WASI spec เป็น Wasm import function และบังคับใช้ capability grant
3. **Wasm Module** — binary ที่คอมไพล์แล้ว มัน import WASI function แบบเดียวกับ Wasm import อื่น ๆ ไม่มีการเรียก OS โดยตรง
